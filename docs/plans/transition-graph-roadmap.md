# Transition graph readability — roadmap

Рабочий план следующих шагов инициативы «читабельность графа переходов».
Ветка: `feat/graph_clarity`. Обновлён: 2026-07-19.

## Сделано (батчи 1–3 + сверх плана)

- **Батч 1, quick wins**: детерминированный рендер, fit окрестности при фокусе,
  убраны молчаливые отсечки, адаптивные подписи рёбер, легенда с индикатором
  покрытия, edge-focus по клику, единый `FIT_PADDING`, мгновенные тултипы.
- **Батч 2, дефолтная читабельность**: бэкенд-layout (hopscotch: word2vec по
  траекториям + вложенные прямоугольники, `tools/graph_layout.py`, зависимость
  gensim), якоря `path_start`/`path_end` по краям, per-node top-k как дефолтный
  Auto-фильтр, лог-шкалы слайдера по диапазону данных, дифф-нормировка толщины.
- **Батч 3, GraphView**: сериализуемые визуальные пресеты
  (focus node/edge/path + direction, фильтры, hiddenEvents, viewport,
  nodePositions); пять входов: kwargs `views=`/`view=`, пиллы, Copy view link,
  `#view=` в экспорте, `[Tab:view=Name]` в MCP.
- **Сверх плана**: направленный node-фокус (клик = исходящие, 2×клик =
  входящие, без цветового кода; дифф — исключение), интерактивный авторинг
  путей (⌘-клик добавляет, повторы дают петли/циклы, ⌘-2×клик — backspace,
  кандидаты от вершины с перевзвешиванием), персистентность ручной
  расстановки по отпечатку данных (`widget_id = "tg-" + fingerprint[:12]`),
  свёрнутая легенда в экспорте, документация Views с демо.

Практика верификации, которая работает: pytest (692) + pre-commit + headless
Chrome поверх экспортов и харнесов с фейковым хостом (симуляция кликов через
`cy.emit`, диагностика через `document.title`). Помнить: живой anywidget-путь
отличается от экспорта (compute!), проверять оба.

## Батч 4 — статистика пути и ego-view

### 4a. Route stats: «этим маршрутом прошло N путей (X%)» — СДЕЛАНО

Реализовано (отклонения от первоначального плана):
- Имя — `route_stats`, не `path_stats` (коллизия: `describe()` уже
  возвращает ключ `path_stats` с другим смыслом; ADR-0008).
- Только **строгое** непрерывное совпадение; вариант со схлопнутыми
  self-loop-заиканиями отложен («начнём со строгого, потом расширим»;
  при намеренной петле в маршруте схлопывание было бы неверным).
- Бэкенд — не path-строки, а lead()-цепочка одним SQL-проходом
  (`tools/route_stats.py`); пересекающиеся вхождения считаются
  (консистентно со счётчиками переходов), `path_start`/`path_end`
  допустимы. Headless-твин `Eventstream.route_stats(nodes, path_col=...)`.
- Метрики: `unique_paths` (+ доля), `occurrences`, `avg_per_path`,
  `time_median`/`time_q95`, `proba` — марковское произведение
  `P(next | source)` (считается на бэке, сверено с TransitionMatrix).
- Бейдж в правом верхнем углу под иконками, селектор метрики; дефолтная
  метрика следует весу рёбер (`proba_in`/`share_of_total` →
  unique paths); существительное только в селекторе, не в значении;
  debounce 400мс, guard от устаревших ответов.

Отложено в бэклог: предпросчёт stats для path-view при
`export_html`/MCP-табах (`view_stats: {viewName: {...}}`) — сейчас в
статическом HTML бейдж скрыт (compute недоступен); замена `proba` на
эмпирическую конверсию «of A visitors: 61% completed the route».

### 4b. Ego-view (модалка) — СДЕЛАНО

Реализовано близко к плану, с уточнением по метрикам весов:
- Отдельный React/SVG-компонент (`EgoView.tsx`), не cytoscape — дублирование
  нод (A слева и справа при A→X и X→A) через обычный SVG, не через
  граф-модель. Вход — кнопка «Ego view» в тулбаре при фокусе на одной ноде;
  оверлей на весь канвас, Esc/фон/крестик — выход; клик по соседу
  перецентровывает view на него (навигация по графу).
- **Отклонение от плана**: стороны показывают не веса графа (текущий
  `edge_weight`), а всегда точные `proba_in` (слева) / `proba_out`
  (справа), посчитанные из сырых счётчиков переходов — иначе смысл ego
  вопроса «откуда сюда приходят» терялся бы при любом другом выбранном
  весе. Потребовало провезти разреженную матрицу `counts` в пейлоаде
  графа независимо от `edge_weight` (работает и в статическом экспорте).
  Фолбэк на веса графа — только в дифф-режиме и для старых экспортов без
  `counts`.
- Центральный бар — компактный (не тянется на всю высоту стека), ленты
  сходятся к нему без зазора (сжатие пропорционально на каждую сторону).
  Хедер — просто «Ego view» с ⓘ-тултипом «Incoming and outgoing edges»
  (без имени ноды — оно и так в центре). Тултип ребра: «A → B X%
  (Y transitions out of Z) – click to recenter». Цвет акцента — янтарный
  (лучше видно на белом, не брендовый жёлтый).
- Не сделано (осталось в бэклоге): `GraphView` v2 адресация
  (`focus: {type: "node", mode: "ego"}`), харнес-скриншоты как формальный
  тест (проверялось вручную через headless Chrome в течение разработки,
  автоматизированного скриншот-теста в repo нет).

## Батч 5 — статистическая значимость

### 5a. Надёжность оценок (обычный режим)

Лечит «67% на трёх наблюдениях»:
- Бэкенд: для `proba_out`/`proba_in` считать поддержку (кол-во переходов)
  на ребро — уже есть count-матрица, отдать парой с вероятностной.
- UI: рёбра с поддержкой ниже порога (или широким интервалом Вильсона)
  приглушать/штриховать; в тултипе ребра показывать CI и поддержку:
  «0.67 (CI 0.21–0.94, n=3)».
- Параметр: `min_support` в сайдбаре (дефолт разумный, например 10).

### 5b. Значимость диффа

- Бэкенд: z-тест двух пропорций на каждое ребро (для proba-типов; для
  count-типов — тест пропорций от исходящих) + поправка Бенджамини–Хохберга
  (рёбер сотни). Результат — матрица p-values рядом с diff-матрицей
  (расширить `transition_graph_data` возврат или отдельный tool).
- UI: незначимые рёбра — серым (не скрывать), тумблер «significant only»,
  уровень значимости в сайдбаре (0.05 дефолт); легенда объясняет серый.
- Это частичное возвращение StatTests из 3.x — зафиксировать ADR-апдейтом
  (ADR-0012 помечал их «may return»).
- Тесты: синтетика с известным ответом (одно ребро с реальным эффектом,
  остальное шум → после FDR значимо только оно).

## Батч 6 — таксономия событий / semantic zoom (отдельный проект)

Идея согласована концептуально: таксономия как **внешний словарь-резолвер**
`{"scroll_submit_form": ["submit_form", "payment_details", "checkout"]}`
(путь в дереве), а не колонка на каждое событие. Требует отдельного дизайна:
- схема: где живёт словарь (`Eventstream`-атрибут? аргумент виджета?),
  влияние на нейминг-конвенции (ADR-0008) → нужен новый ADR;
- UI: группировка нод в мета-ноды по уровню таксономии, переключатель
  уровня («zoom») — compound nodes cytoscape или пересчёт матрицы по
  уровню (вероятнее второе: агрегированная матрица честнее);
- MCP: уровень как параметр `add_transition_graph`.

Не начинать до мержа батчей 4–5.

## Бэклог (мелочи, брать между батчами)

- **Фильтр диффа по знаку**: тумблер red/blue/all рядом с Auto/Manual —
  «показать только то, что выросло».
- **Галерея эталонных датасетов** как eval-harness: расширить
  `generate_widget_demos.py` наборами маленький/большой/плотный/сессионный/
  дифф — гонять визуальные изменения по всем (пригодится в 4–5).
- **`multiClickDebounceTime`**: если 250мс-окно двойного клика будет мешать
  быстрому набору петель (`B→B`) — сузить.
- **Hash-view в report-шаблоне**: сейчас `#view=` работает только в
  bare-экспорте; в отчёте нужен `#tab=...&view=...`.
- **`focus.type: "nodes"`** (неупорядоченный набор) — если появится
  реальная потребность, машинерия мультифокуса сохранена.
- **Правая кнопка/контекстное меню** на ноде — кандидат на «удалить из
  пути в середине» и другие действия, если backspace-семантики не хватит.
- **Известные ограничения** (задокументировать в guide, если будут жалобы):
  расстановка привязана к точному состоянию данных (fingerprint включает
  счётчики); окно двойного клика 250мс; MCP-оверлей и интерактивный путь —
  разные механизмы подсветки.
