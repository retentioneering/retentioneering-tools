# ADR-0014: Docstring-driven documentation pipeline

Status: Accepted (recorded 2026-07)

## Context

Hand-written reference docs drift from the code, and this library has three
consumers of the same reference material: the docs site, Python `help()`,
and LLM agents (the MCP server serves method docstrings verbatim via
`describe_tool`).

## Decision

- **Docstrings are the single source of truth for reference docs.** Widget
  and data-processor pages are rendered from `Eventstream` docstrings by
  `docs/scripts/render_pages.py` through per-page Jinja templates
  (`docs/templates/`); a page's shape lives in its template, its content in
  the docstring.
- Widget pages split parameters into Data / Display groups mechanically, by
  diffing the widget signature against its headless twin (ADR-0006) —
  the split cannot drift.
- Hand-written conceptual pages (`docs/guide/*.md`: quick start, Eventstream
  guide with the "Key concepts" glossary, Path Metrics) are copied as-is.
- Live-looking widget demos are generated by
  `docs/scripts/generate_widget_demos.py`: every `<DemoWidget cmd={...}>`
  tag in templates/guides is executed against the bundled ecom dataset and
  exported via the real `export_html()` (ADR-0011), so docs examples are
  continuously executed code.
- The built docs live in `docs/build/` and are served by the separate
  retentioneering-web project.

## Consequences

- Changing a public signature means changing its docstring, which *is* the
  documentation change; then re-run `render_pages.py` (and
  `generate_widget_demos.py` if examples changed).
- Docstrings must be written for three readers at once — humans on the docs
  site, `help()`, and LLMs — hence complete enum listings and runnable
  examples (ADR-0008).
- A demo that stops constructing (e.g. after a rename) breaks demo
  generation loudly — an extra integration check the test suite doesn't
  provide for widgets.
